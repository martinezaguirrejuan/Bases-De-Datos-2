INNER JOIN doble — Ventas con cliente y empleado

ventas · clientes · empleados
JOIN
Tablas

ventas (central) → clientes (FK cedula_cliente) → empleados (FK cedula_empleado)

Qué hace

Reemplaza las cédulas por los nombres reales de cliente y empleado en cada venta.

Resultado esperado

9 filas — excluye la venta #8 porque su cliente es NULL (anónima).

Por qué es importante
Es el patrón fundamental del modelo: ventas solo guarda claves foráneas; para leer nombres siempre hay que hacer JOIN. INNER JOIN descarta automáticamente las filas sin coincidencia (NULL), por eso no aparece la venta anónima.
SQL 
SELECT v.id_venta,
       c.nombre         AS cliente,
       e.nombre         AS empleado,
       v.fecha_venta,
       v.total_venta,
       v.metodo_pago
FROM ventas v
INNER JOIN clientes  c ON v.cedula_cliente  = c.cedula
INNER JOIN empleados e ON v.cedula_empleado = e.cedula;

LEFT JOIN + COALESCE — Todas las ventas (incluye anónimas)

ventas · clientes (LEFT) · empleados
JOIN
Tablas

ventas → LEFT JOIN clientes (puede ser NULL) + INNER JOIN empleados

Qué hace

Trae las 10 ventas. COALESCE reemplaza NULL por "Cliente Anónimo".

Resultado esperado

10 filas — venta #8 aparece como "Cliente Anónimo".

Por qué es importante
Diferencia clave: INNER JOIN descarta nulos, LEFT JOIN los conserva. COALESCE(valor, 'default') es el patrón estándar para sustituir NULL por un texto legible. En ventas reales siempre habrá compras sin cliente registrado.
SQL 
SELECT v.id_venta,
       COALESCE(c.nombre, 'Cliente Anónimo') AS cliente,
       e.nombre                               AS empleado,
       v.total_venta,
       v.metodo_pago
FROM ventas v
LEFT  JOIN clientes  c ON v.cedula_cliente  = c.cedula
INNER JOIN empleados e ON v.cedula_empleado = e.cedula;

LEFT JOIN + IS NULL — Clientes que NUNCA han comprado

clientes · ventas (LEFT)
Anti-Join
Tablas

clientes (maestra) + LEFT JOIN ventas. Filas sin venta → v.id_venta = NULL.

Qué hace

Identifica clientes registrados sin ninguna venta asociada.

Resultado esperado

Clientes sin historial — candidatos a campañas de activación.

Por qué es importante
Patrón "anti-join con LEFT JOIN": más eficiente que NOT IN en tablas grandes. LEFT JOIN trae todos los clientes; WHERE v.id_venta IS NULL filtra solo los que no tienen fila coincidente en ventas.
SQL 
SELECT c.cedula, c.nombre, c.email
FROM clientes c
LEFT JOIN ventas v ON c.cedula = v.cedula_cliente
WHERE v.id_venta IS NULL;

JOIN triple + GROUP BY — Ingresos por tipo de licor

productos · detalle_ventas · ventas
GROUP BY
Tablas

Cadena A→B→C: productos → detalle_ventas → ventas

Qué hace

Calcula unidades e ingresos por tipo de licor en ventas completadas.

Resultado esperado

Una fila por tipo_licor con totales, ordenada de mayor a menor ingreso.

Por qué es importante
Demuestra el JOIN en cadena (tres tablas). El WHERE sobre ventas.estado filtra datos antes de agrupar; si se pusiera en HAVING, filtraría grupos ya formados (resultado diferente). Este reporte identifica qué categoría genera más dinero.
SQL 
SELECT p.tipo_licor,
       SUM(dv.cantidad)  AS unidades_vendidas,
       SUM(dv.subtotal)  AS ingresos_totales
FROM productos p
JOIN detalle_ventas dv ON p.id_producto = dv.id_producto
JOIN ventas v          ON dv.id_venta   = v.id_venta
WHERE v.estado = 'Completada'
GROUP BY p.tipo_licor
ORDER BY ingresos_totales DESC;

GROUP BY + HAVING — Empleados con ventas > $200.000

empleados · ventas
HAVING
Tablas

empleados + JOIN ventas, agrupados por empleado.

Qué hace

Filtra solo empleados cuyo monto total de ventas supera $200.000.

Resultado esperado

Solo los empleados de alto desempeño — los que más venden.

Por qué es importante
Diferencia fundamental WHERE vs HAVING: WHERE filtra filas antes de agrupar; HAVING filtra grupos después. No se puede escribir WHERE SUM(...) — es error de sintaxis. HAVING es el único lugar válido para condiciones sobre funciones de agregado.
SQL 
SELECT e.nombre, e.cargo,
       COUNT(v.id_venta)   AS cant_ventas,
       SUM(v.total_venta)  AS total_vendido
FROM empleados e
JOIN ventas v ON e.cedula = v.cedula_empleado
GROUP BY e.cedula, e.nombre, e.cargo
HAVING SUM(v.total_venta) > 200000
ORDER BY total_vendido DESC;

Subconsulta escalar — Productos sobre el precio promedio

productos (exterior + interior)
Subconsulta
Tipo de subconsulta

Escalar: devuelve exactamente 1 fila × 1 columna (el promedio).

Qué hace

Calcula AVG internamente y lo usa como umbral para filtrar productos caros.

Resultado esperado

~5 productos premium: Buchanans, Santa Teresa, Old Parr, Cuervo, Absolut.

Por qué es importante
La subconsulta escalar se ejecuta una vez; el motor guarda el resultado y lo compara con cada fila de la consulta exterior. Permite filtrar con valores calculados dinámicamente sin necesidad de conocer el promedio de antemano.
SQL 
SELECT nombre, tipo_licor, precio_venta
FROM productos
WHERE precio_venta > (SELECT AVG(precio_venta) FROM productos)
ORDER BY precio_venta DESC;

Subconsulta en FROM (tabla derivada) — Empleados con >1 venta

empleados · ventas (JOIN dentro de la subconsulta)
Tabla derivada
Tipo de subconsulta

En FROM: crea una tabla virtual "emp_ventas" con el conteo por empleado.

Qué hace

Filtra los empleados que procesaron más de una venta.

Resultado esperado

Laura Martínez (3), Diego Castro (2), Sandra Rojas (2).

Por qué es importante
Permite aplicar un WHERE sobre el resultado de un GROUP BY, algo imposible directamente. El alias AS emp_ventas es obligatorio en MariaDB. Más legible que anidar HAVING cuando la lógica exterior es compleja.
SQL 
SELECT emp_ventas.nombre, emp_ventas.cargo, emp_ventas.total_ventas
FROM (
    SELECT e.nombre, e.cargo, COUNT(v.id_venta) AS total_ventas
    FROM empleados e
    JOIN ventas v ON e.cedula = v.cedula_empleado
    GROUP BY e.cedula, e.nombre, e.cargo
) AS emp_ventas
WHERE emp_ventas.total_ventas > 1
ORDER BY emp_ventas.total_ventas DESC;

DELETE + NOT IN — Eliminar clientes sin historial

clientes · ventas (subconsulta)
DELETE
Tablas

DELETE en clientes; subconsulta obtiene cédulas de ventas.

Qué hace

Elimina clientes que no tienen ninguna venta registrada.

Trampa importante

Sin IS NOT NULL en la subconsulta, un NULL anularía NOT IN para todas las filas.

Por qué es importante
El bug clásico de NOT IN: si la subconsulta devuelve algún NULL, la expresión X NOT IN (..., NULL, ...) siempre es UNKNOWN (nunca TRUE) y no se elimina nada. El WHERE cedula_cliente IS NOT NULL es la salvaguarda obligatoria.
SQL 
DELETE FROM clientes
WHERE cedula NOT IN (
    SELECT DISTINCT cedula_cliente
    FROM ventas
    WHERE cedula_cliente IS NOT NULL
);

UPDATE con JOIN — Descuento 10% en cervezas

detalle_ventas · productos
UPDATE + JOIN
Tablas

detalle_ventas (se modifica) + JOIN productos (fuente del filtro tipo_licor).

Qué hace

Aplica 10% de descuento y recalcula subtotal en líneas de Cerveza.

Resultado esperado

Cerveza Poker y Corona quedan con descuento y subtotal al 90%.

Por qué es importante
UPDATE con JOIN es exclusivo de MySQL/MariaDB (no estándar SQL). Permite modificar una tabla usando datos de otra sin subconsultas. ROUND(..., 2) es obligatorio en operaciones monetarias para evitar errores de punto flotante.
SQL 
UPDATE detalle_ventas dv
    JOIN productos p ON dv.id_producto = p.id_producto
    SET dv.descuento = ROUND(dv.precio_unitario * 0.10, 2),
        dv.subtotal  = ROUND(dv.cantidad * dv.precio_unitario * 0.90, 2)
WHERE p.tipo_licor = 'Cerveza';

UPDATE en cascada (JOIN triple) — Cancelar venta y reponer stock

ventas · detalle_ventas · productos
Cascada
Tablas modificadas

ventas (estado → Cancelada) Y productos (stock_actual + cantidad devuelta).

Qué hace

Cancela la venta y repone inventario en una sola sentencia.

Resultado esperado

Venta #7 = Cancelada; Aguardiente Antioqueño recupera sus unidades.

Por qué es importante
Un UPDATE modifica dos tablas distintas al mismo tiempo. Sin esta técnica habría que lanzar dos sentencias separadas con riesgo de inconsistencia si la segunda falla. El JOIN triple v→dv→p es necesario para "llegar" a productos partiendo del id_venta.
SQL 
UPDATE ventas v
    JOIN detalle_ventas dv ON v.id_venta     = dv.id_venta
    JOIN productos p       ON dv.id_producto = p.id_producto
    SET v.estado       = 'Cancelada',
        p.stock_actual = p.stock_actual + dv.cantidad
WHERE v.id_venta = 7;

ENUM + SET + FIND_IN_SET — Vendedores sin miércoles

empleados
ENUM / SET
Tipo ENUM

cargo: valor único de lista predefinida. Se filtra con = o IN por texto o índice numérico.

Tipo SET

dias_labora: múltiples valores seleccionables. Se busca con FIND_IN_SET().

Resultado esperado

Vendedores cuyo horario no incluye el miércoles.

Por qué es importante
ENUM almacena un valor único de un dominio fijo; SET permite múltiples valores combinados. Para buscar dentro de un SET no se puede usar = (compara el SET completo); FIND_IN_SET('valor', columna) es la función correcta. NOT FIND_IN_SET verifica ausencia del valor.
SQL 
SELECT nombre, cargo, dias_labora
FROM empleados
WHERE cargo IN ('Vendedor', 'Vendedora')
  AND NOT FIND_IN_SET('miercoles', dias_labora);

INSERT INTO … SELECT — Registrar venta por nombre de empleado

ventas (destino) · empleados (fuente)
INSERT SELECT
Tablas

INSERT en ventas; SELECT busca la cédula en empleados por nombre.

Qué hace

Inserta una venta sin necesidad de saber la cédula del empleado de memoria.

Resultado esperado

Nueva venta con la cédula de Laura Martínez obtenida automáticamente.

Por qué es importante
INSERT INTO … SELECT permite insertar datos derivados de otras tablas. Si el nombre no existe en empleados, el INSERT devuelve 0 filas afectadas (no falla, simplemente no inserta nada) — hay que verificarlo en la aplicación.
SQL 
INSERT INTO ventas (cedula_cliente, cedula_empleado, total_venta, metodo_pago, estado)
    SELECT '3344556677', cedula, 56000, 'Efectivo', 'Completada'
    FROM empleados
    WHERE nombre = 'Laura Martínez Pérez';

Variable de usuario — Empleado con más ventas (2 pasos)

ventas (paso 1) · empleados (paso 2)
@variable
Paso 1

Guarda en @top_cedula la cédula del empleado con mayor COUNT de ventas.

Paso 2

Usa @top_cedula para recuperar los datos completos del empleado.

Resultado esperado

@top_cedula = 43219876 → Laura Martínez Pérez, Vendedora.

Por qué es importante
Las variables @nombre persisten durante toda la sesión. Son más legibles que anidar subconsultas complejas. Clave: la variable solo existe mientras dure la conexión con el servidor; no se puede usar entre sesiones distintas.
SQL 
-- Paso 1: guardar la cédula
SELECT @top_cedula := cedula_empleado
FROM ventas
GROUP BY cedula_empleado
ORDER BY COUNT(*) DESC
LIMIT 1;

-- Paso 2: usar la variable
SELECT cedula, nombre, cargo, salario
FROM empleados
WHERE cedula = @top_cedula;

Encriptación SHA1 — Verificar login de usuario

usuarios_licoreria
SHA1 / MD5
SHA1

Produce 40 caracteres hex. MD5 produce 32. Ambos son irreversibles.

Qué hace

Hashea la clave ingresada y compara con el hash almacenado. Nunca compara texto plano.

Resultado

Clave correcta → 1 fila. Clave incorrecta → Empty set (acceso denegado).

Por qué es importante
Los hashes son IRREVERSIBLES — no se puede recuperar la clave original. El sistema hashea el input del usuario y compara hashes; así aunque alguien acceda a la BD no conoce las contraseñas reales. SHA1 ≠ MD5: producen hashes de distinta longitud y no son compatibles entre sí.
SQL 
-- Login exitoso
SELECT usuario, clave
FROM usuarios_licoreria
WHERE usuario = 'gerente' AND clave = SHA1('Abc123');

-- Login fallido (clave incorrecta → Empty set)
SELECT usuario, clave
FROM usuarios_licoreria
WHERE usuario = 'gerente' AND clave = SHA1('XYZ999');
Comparación de algoritmos
SHA1('Abc123') → 40 chars hex  |  MD5('Abc123') → 32 chars hex  |  PASSWORD('Abc123') → 41 chars

Trigger descontar_stock — Automatizar inventario

AFTER INSERT ON detalle_ventas → productos
TRIGGER
Evento

AFTER INSERT en detalle_ventas: se dispara tras insertar una línea de venta.

Qué hace

Descuenta automáticamente NEW.cantidad del stock_actual del producto vendido.

Resultado

Inventario siempre actualizado sin intervención manual en cada venta.

Por qué es importante
NEW hace referencia a la fila recién insertada. AFTER INSERT garantiza que el INSERT ya fue confirmado antes de ejecutar el UPDATE al stock. Si el UPDATE del trigger falla, MariaDB puede revertir ambas operaciones manteniendo consistencia entre ventas e inventario.
SQL 
DELIMITER ;;
CREATE TRIGGER descontar_stock
AFTER INSERT ON detalle_ventas
FOR EACH ROW
BEGIN
  UPDATE productos
  SET stock_actual = stock_actual - NEW.cantidad
  WHERE id_producto = NEW.id_producto;
END;;
DELIMITER ;

-- Verificar que el trigger existe:
SHOW TRIGGERS LIKE 'descontar_stock';
