Relaciones entre Entidades

Cardinalidades, llaves foráneas y comportamiento referencial

👤 CLIENTES → 🛒 VENTAS 1 : N

Realiza — Un cliente puede tener muchas ventas. Una venta pertenece a un solo cliente (o es anónima).

ventas.cedula_cliente → clientes.cedula
ON DELETE CASCADE
🧑‍💼 EMPLEADOS → 🛒 VENTAS 1 : N

Atiende — Un empleado puede atender muchas ventas. Cada venta es atendida por un solo empleado.

ventas.cedula_empleado → empleados.cedula
ON DELETE CASCADE
🛒 VENTAS ↔ 🍾 PRODUCTOS N : M

Contiene / es vendido en — Resuelta con la tabla intermedia detalle_ventas.

detalle_ventas.id_venta → ventas.id_venta
detalle_ventas.id_producto → productos.id_producto
🏭 PROVEEDORES → 📦 COMPRAS 1 : N

Suministra en — Un proveedor puede tener muchas compras. Cada compra pertenece a un solo proveedor.

compras.nit_proveedor → proveedores.nit
ON DELETE CASCADE
📦 COMPRAS ↔ 🍾 PRODUCTOS N : M

Contiene / es comprado en — Resuelta con la tabla intermedia detalle_compras.

detalle_compras.id_compra → compras.id_compra
detalle_compras.id_producto → productos.id_producto

Tras cada INSERT en detalle_ventas, el trigger descontar_stock descuenta automáticamente el inventario de PRODUCTOS.

Trigger: descontar_stock

Automatización del inventario tras cada venta registrada

Nombredescontar_stock
EventoAFTER INSERT
Tabladetalle_ventas
PropósitoConsistencia automática de inventario

Código SQL

CREATE TRIGGER descontar_stock
AFTER INSERT ON detalle_ventas
FOR EACH ROW
BEGIN
    UPDATE productos
    SET stock_actual = stock_actual - NEW.cantidad
    WHERE id_producto = NEW.id_producto;
END;

Encriptación — SHA1, MD5 y PASSWORD

Protección de contraseñas mediante funciones de hashing irreversible en MariaDB

🔐 Funciones de encriptación en MariaDB

Comparativa de los tres algoritmos de hash usados para proteger contraseñas

FunciónAlgoritmoLongitud resultadoReversibleDescripción
SHA1(str) SHA-1 (160 bits) 40 chars hex No Secure Hash Algorithm 1. Produce un resumen del mensaje de 40 dígitos hexadecimales.
MD5(str) MD5 (128 bits) 32 chars hex No Message-Digest Algorithm 5. Hash de 32 caracteres hex. Ampliamente usado.
PASSWORD(str) Hash interno MariaDB 41 chars (*XXX…) No Función usada por MariaDB para cifrar contraseñas de sus usuarios del sistema (tabla mysql.user).

🍾 Tabla: usuarios_licoreria

Gestiona el acceso al sistema. Las claves se almacenan como hash — nunca como texto plano.

CampoTipoNuloClaveDescripción
idVARCHAR(5)NOPKIdentificador único del usuario del sistema
usuarioVARCHAR(20)NO—Nombre de usuario para iniciar sesión
claveVARCHAR(41)NO—Hash de la contraseña. VARCHAR(41) cubre los 3 algoritmos: SHA1=40, MD5=32, PASSWORD=41
estadoVARCHAR(5)NO—Estado de la cuenta: A (Activo) / I (Inactivo)

Paso 1 — Ver cómo lucen los hashes

SELECT SHA1('Abc123')     AS hash_sha1;
-- Resultado: bec75d2e4e2acf4f4ab038144c0d862505e52d07  (40 chars)

SELECT MD5('Abc123')      AS hash_md5;
-- Resultado: 61bd60c60d9fb60cc8fc7767669d40a1          (32 chars)

SELECT PASSWORD('Abc123') AS hash_password;
-- Resultado: *7A8BBCB18A250055A6BB98ECFA33A8174D21950  (41 chars)

Paso 2 — Crear la tabla de usuarios

CREATE TABLE usuarios_licoreria (
  id      VARCHAR(5)  NOT NULL,
  usuario VARCHAR(20) NOT NULL,
  clave   VARCHAR(41) NOT NULL,   -- 41 chars cubre los 3 algoritmos
  estado  VARCHAR(5)  NOT NULL,
  PRIMARY KEY (id)
);

Paso 3 — Insertar con contraseñas cifradas al momento del INSERT

INSERT INTO usuarios_licoreria VALUES ('00001', 'gerente',  SHA1('Abc123'),     'A');
INSERT INTO usuarios_licoreria VALUES ('00002', 'vendedor', MD5('Abc123'),      'A');
INSERT INTO usuarios_licoreria VALUES ('00003', 'cajero',   PASSWORD('Abc123'), 'A');

SELECT * FROM usuarios_licoreria;
-- +-------+----------+------------------------------------------+--------+
-- | 00001 | gerente  | bec75d2e4e2acf4f4ab038144c0d862505e52d07 | A      |
-- | 00002 | vendedor | 61bd60c60d9fb60cc8fc7767669d40a1         | A      |
-- | 00003 | cajero   | *7A8BBCB18A250055A6BB98ECFA33A8174D21950 | A      |
-- +-------+----------+------------------------------------------+--------+

Paso 4 — Verificar login (simular autenticación)

-- Se hashea la clave ingresada y se compara con la almacenada

-- Login correcto con SHA1
SELECT usuario, clave FROM usuarios_licoreria
WHERE usuario = 'gerente' AND clave = SHA1('Abc123');
-- Resultado: devuelve el registro ✔

-- Login correcto con MD5
SELECT usuario, clave FROM usuarios_licoreria
WHERE usuario = 'vendedor' AND clave = MD5('Abc123');
-- Resultado: devuelve el registro ✔

-- Clave INCORRECTA — Empty set (acceso denegado)
SELECT usuario, clave FROM usuarios_licoreria
WHERE usuario = 'gerente' AND clave = SHA1('XYZ999');
-- Resultado: Empty set ✘

Paso 5 — Comparar los tres algoritmos

SELECT 'SHA1'     AS algoritmo, SHA1('Abc123'),     LENGTH(SHA1('Abc123'))     AS chars
UNION ALL
SELECT 'MD5',                   MD5('Abc123'),      LENGTH(MD5('Abc123'))
UNION ALL
SELECT 'PASSWORD',              PASSWORD('Abc123'), LENGTH(PASSWORD('Abc123'));
-- SHA1     → 40 chars
-- MD5      → 32 chars
-- PASSWORD → 41 chars
🔒 Hash irreversible

SHA1 y MD5 convierten un texto en una huella digital de longitud fija. No es posible recuperar la contraseña original a partir del hash almacenado.

SHA1('Abc123') → bec75d2e... (siempre igual)
✅ Verificación de contraseña

Para verificar una clave se hashea el valor ingresado y se compara con el almacenado. Si coinciden, la contraseña es correcta — sin revelar el texto plano.

WHERE clave = SHA1('clave_ingresada')
📏 VARCHAR(41) en columna Clave

SHA1 produce 40 chars, MD5 produce 32 chars y PASSWORD produce 41 chars. Se usa VARCHAR(41) para cubrir los tres algoritmos en la misma columna.

clave VARCHAR(41) — cubre SHA1, MD5 y PASSWORD

Administración de Usuarios y Concepto de HOST

Gestión de cuentas, privilegios y el esquema de seguridad usuario@host de MariaDB

🌐 ¿Qué es el HOST en MySQL/MariaDB?

En MariaDB un usuario se identifica siempre por la combinación usuario + host. El host indica desde dónde se conecta ese usuario. Esto permite que el mismo nombre tenga permisos distintos según el origen de la conexión.


Ejemplo: root@localhost, root@127.0.0.1 y root@::1 son tres cuentas diferentes aunque tengan el mismo nombre de usuario.

root@localhost → acceso solo desde la máquina local (socket Unix)
root@127.0.0.1 → acceso desde IPv4 loopback
root@::1 → acceso desde IPv6 loopback
usuario@% → % significa cualquier host (acceso remoto)

👥 Usuarios creados para la base de datos licoreria

Tres roles con permisos diferenciados según su función

UsuarioHostContraseñaPermisos asignadosRol
gerente_lic localhost gerente123 ALL PRIVILEGES ON licoreria.* Administrador completo de la licorería
vendedor_lic localhost vendedor123 SELECT, INSERT en ventas y detalle_ventas; SELECT en clientes y productos Puede registrar ventas y consultar catálogo
consultor_lic localhost (eliminado) SELECT ON licoreria.* (fue revocado y el usuario eliminado) Solo lectura — eliminado en la demostración

Verificar usuario actual y listar usuarios existentes

SELECT USER();                          -- muestra: root@localhost
SELECT User, Host FROM mysql.user       -- lista completa de cuentas
ORDER BY User;                          -- ordenado por nombre

Crear usuarios con host específico

-- La sintaxis siempre es: 'usuario'@'host'
CREATE USER 'gerente_lic'@'localhost'  IDENTIFIED BY 'gerente123';
CREATE USER 'vendedor_lic'@'localhost' IDENTIFIED BY 'vendedor123';
CREATE USER 'consultor_lic'@'localhost' IDENTIFIED BY 'consultor123';

Asignar permisos (GRANT)

-- Todos los privilegios en la base de datos licoreria
GRANT ALL PRIVILEGES ON licoreria.* TO 'gerente_lic'@'localhost';

-- Permisos específicos solo sobre tablas de ventas
GRANT SELECT, INSERT, UPDATE ON licoreria.ventas         TO 'vendedor_lic'@'localhost';
GRANT SELECT, INSERT, UPDATE ON licoreria.detalle_ventas TO 'vendedor_lic'@'localhost';
GRANT SELECT                 ON licoreria.clientes        TO 'vendedor_lic'@'localhost';
GRANT SELECT                 ON licoreria.productos       TO 'vendedor_lic'@'localhost';

-- Solo lectura sobre toda la base de datos
GRANT SELECT ON licoreria.* TO 'consultor_lic'@'localhost';

FLUSH PRIVILEGES;   -- aplica los cambios inmediatamente

Ver los permisos de un usuario (SHOW GRANTS)

SHOW GRANTS FOR 'gerente_lic'@'localhost';
-- Resultado:
-- GRANT ALL PRIVILEGES ON `licoreria`.* TO 'gerente_lic'@'localhost'

SHOW GRANTS FOR 'vendedor_lic'@'localhost';
-- Resultado:
-- GRANT SELECT, INSERT, UPDATE ON `licoreria`.`ventas` TO 'vendedor_lic'@'localhost'
-- GRANT SELECT, INSERT, UPDATE ON `licoreria`.`detalle_ventas` TO ...

Revocar permisos específicos (REVOKE)

-- Quitar permiso de actualizar ventas al vendedor
REVOKE UPDATE ON licoreria.ventas         FROM 'vendedor_lic'@'localhost';
REVOKE UPDATE ON licoreria.detalle_ventas FROM 'vendedor_lic'@'localhost';

-- Verificar que UPDATE fue removido
SHOW GRANTS FOR 'vendedor_lic'@'localhost';

Modificar contraseña y eliminar usuario

-- Cambiar la contraseña del consultor
SET PASSWORD FOR 'consultor_lic'@'localhost' = PASSWORD('nueva_clave789');

-- Eliminar el usuario consultor
DROP USER 'consultor_lic'@'localhost';

-- Verificar que fue eliminado
SELECT User, Host FROM mysql.user ORDER BY User;
🔑 GRANT — Asignar permisos

Otorga privilegios a un usuario. Se puede aplicar a todas las BDs (*.*), a una BD completa (licoreria.*) o a una tabla específica (licoreria.ventas).

GRANT permiso ON bd.tabla TO 'usuario'@'host';
🚫 REVOKE — Revocar permisos

Elimina un permiso específico que fue otorgado previamente. No elimina el usuario, solo le quita el acceso al recurso indicado.

REVOKE permiso ON bd.tabla FROM 'usuario'@'host';
🗑 DROP USER — Eliminar usuario

Borra permanentemente la cuenta de usuario del sistema. Automáticamente revoca todos sus permisos. Se debe especificar el host exacto.

DROP USER 'usuario'@'host';
🔄 FLUSH PRIVILEGES

Recarga las tablas de privilegios de la base de datos mysql para que los cambios de GRANT y REVOKE surtan efecto de forma inmediata en la sesión.

FLUSH PRIVILEGES; — ejecutar tras cambios de permisos
